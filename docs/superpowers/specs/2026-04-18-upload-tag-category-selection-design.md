# 投稿页分类与标签选择体验优化设计

## 1. 背景

当前投稿页的分类与标签选择已经具备基础能力：

- 分类区域已经支持一级 / 二级分类切换
- 标签区域已经支持手动选择、搜索标签、推荐标签
- 投稿页整体视觉风格已经比较稳定

但当前体验仍有几个明显问题：

1. 标签平铺过多，初始认知负担偏高
2. 分类虽然已有二级结构，但“当前是否选完”不够直观
3. 分类与标签之间缺少明显联动，用户选完分类后仍然面对一整屏无差别标签
4. 推荐标签目前带有“AI”心智，但当前实现实际上是本地规则推荐，不适合继续用 AI 表述

本次设计目标不是重做投稿页，而是在**不改变整体 UI 风格**的前提下，对分类与标签选择流程做渐进增强。

---

## 2. 设计目标

### 2.1 目标

1. 保持当前投稿页整体布局、卡片样式、标签 chip 风格不变
2. 让分类选择状态更明确，降低“我到底选没选好”的困惑
3. 让标签选择从“全量平铺”变成“有层次地逐步筛选”
4. 建立分类与标签之间的联动，让标签选择更聚焦
5. 保留自动推荐标签能力，但移除“AI”表述

### 2.2 非目标

1. 不重做投稿页的左右双栏布局
2. 不引入新的视觉风格或大规模 UI 重绘
3. 不引入外部 AI 服务或大模型依赖
4. 不在本阶段重做数据库标签体系

---

## 3. 已确认的产品决策

### 3.1 分类规则

分类选择规则调整为：

- 一级分类可以直接作为最终分类
- 如果某一级下存在二级分类，界面应**优先引导**用户继续选择二级
- 但若用户认为二级都不合适，允许直接使用一级分类提交
- 如果用户选择了二级分类，则最终以二级分类作为 `categoryId`

即：**二级优先，但一级可兜底**。

### 3.2 标签规则

- 保留自动推荐标签能力
- 取消“AI 推荐”表述
- 推荐标签本质定义为：基于标题、简介和已有规则的本地智能推荐

### 3.3 分类与标签联动

- 用户选中分类后，标签区应优先展示该分类相关标签
- 相关标签展示优先级高于全量标签
- 全量标签仍可访问，但不应作为默认第一视图

### 3.4 视觉约束

- 保持现有 UI 风格不变
- 延续现有卡片、圆角、浅色背景、tag chip 风格
- 本次优化重点是信息结构和交互层次，而不是视觉重设计

---

## 4. 方案对比

### 方案 A：渐进改版（选中方案）

在现有投稿页结构上增强分类和标签交互：

- 分类区保留现有一级 / 二级结构，但增强状态提示和已选路径显示
- 标签区拆成“已选标签 / 智能推荐 / 分类相关 / 更多标签”四层
- 全量标签不再默认大面积平铺，而是折叠到“更多标签”中

**优点：**

- 风险低，容易落地
- 与现有页面结构兼容性高
- 不改变用户对现有页面的整体认知
- 最符合“UI 风格不变”的要求

**缺点：**

- 本质是增强，不是彻底重构
- “更多标签”的组织方式仍需要额外设计

### 方案 B：引导式流程重排

将投稿页右侧做成更明显的“先分类、再标签”的流程化结构。

**优点：**

- 认知路径更清晰
- 更符合投稿时的自然顺序

**缺点：**

- 改动较大
- 更容易触碰当前页面风格与布局稳定性

### 方案 C：强层级结构化重做

把标签和分类都做成更强的树状或面板式选择器。

**优点：**

- 结构感最强

**缺点：**

- 复杂度过高
- 不适合当前项目阶段
- 会明显改变页面调性

### 结论

选择**方案 A：渐进改版**。

---

## 5. 分类区设计

当前分类区域保留现有“一级分类 + 二级分类”基本结构，但补充更明确的状态表达。

### 5.1 分类区展示结构

分类卡片内保留两层：

1. 一级分类区
2. 二级分类区（仅在当前一级存在子类时展示）

### 5.2 分类状态模型

分类区存在三种核心状态：

#### 状态 A：未选择任何分类

顶部状态文案：

- `请选择分类`

#### 状态 B：已选择一级分类，但未选择二级分类

若该一级下存在二级分类，则顶部状态文案：

- `当前分类：科技`
- 辅助提示：`可继续细分到二级分类，也可直接使用一级分类`

若该一级下不存在二级分类，则顶部状态文案：

- `已选分类：Vlog`

#### 状态 C：已选择二级分类

顶部状态文案：

- `已选分类：科技 / 后端开发`

### 5.3 一级分类交互

点击一级分类后：

- 若该一级分类无子类：
  - 直接把该一级分类设为最终分类
- 若该一级分类有子类：
  - 先把该一级置为当前活动分类
  - 默认将当前分类状态记为“一级已选”
  - 展示其二级分类列表
  - 不强制用户必须选二级，但用提示语引导继续细分

### 5.4 二级分类交互

点击二级分类后：

- 将该二级分类设为最终分类
- 顶部状态同步变为完整路径

### 5.5 分类区设计原则

1. 一级分类永远可兜底，不把用户卡死在二级选择上
2. 有二级时仍然优先鼓励细分，保证分类数据尽量精确
3. 不通过红色报错来做主引导，而是通过温和提示和已选状态让用户理解当前进度

---

## 6. 标签区设计

标签区是本次优化的重点。当前平铺式选择将被替换为分层选择结构。

### 6.1 标签区新的信息层次

标签卡片内部按以下顺序组织：

1. 已选标签
2. 智能推荐标签
3. 当前分类相关标签
4. 更多标签
5. 搜索标签输入框

说明：搜索框仍然保留，但不再承担唯一入口，而是作为补充入口。

### 6.2 已选标签区

保持当前设计思路：

- 放在标签卡片最上方
- 已选标签以高亮 chip 展示
- 支持逐个取消和一键清空

作用：

- 让用户随时知道自己已经选了什么
- 减少重复选择和遗漏感

### 6.3 智能推荐标签区

当前“AI 推荐”语义改为更准确的文案。

推荐文案候选：

- `智能推荐`
- `推荐标签`
- `根据内容推荐`

本设计推荐使用：

- 区块标题：`智能推荐`
- 按钮文案：`智能推荐`

说明文案不强调 AI，而强调“根据内容自动给建议”。

该区域来源仍然使用现有本地规则能力，不引入外部模型。

### 6.4 当前分类相关标签区

这是新增层。

用户一旦选择分类，标签区就优先展示该分类相关的标签。

示例：

- 选择 `科技`
  - 优先展示：Java、SpringBoot、Redis、MySQL、Docker、人工智能、计算机网络
- 选择 `科技 / 后端开发`
  - 更聚焦展示：Java、SpringBoot、MySQL、Redis、Docker、项目实战、接口开发

该层的目标不是限制标签，而是：

- 优先提供更可能被选中的标签
- 降低用户在全量标签中扫视的成本

### 6.5 更多标签区

更多标签作为折叠区保留，默认不全部展开。

展开后，标签按主题分组展示。

这里的“主题分组”只是 UI 组织方式，不代表标签本身一定要改成数据库层级树结构。

推荐分组示例：

- 编程语言
- 框架 / 中间件
- 数据库 / 缓存
- 内容形式
- 方向 / 场景

用户可以跨组多选标签，标签本身仍保持平级语义。

### 6.6 搜索标签

搜索标签输入框继续保留，但位置放在“更多标签”入口附近或其上方。

原则：

- 搜索是补充手段，不是默认主路径
- 默认主路径是：看推荐 -> 看分类相关 -> 再展开更多 -> 不够再搜索

---

## 7. 标签推荐与联动逻辑

### 7.1 推荐标签来源

推荐标签不使用外部 AI，继续沿用当前本地规则推荐能力。

推荐信号可来自：

1. 标题命中
2. 简介命中
3. 分类联想
4. 常见共现标签（如有）

当前阶段至少保证前 3 项成立。

### 7.2 分类联动优先级

标签区默认展示优先级定义为：

1. 已选标签
2. 智能推荐标签
3. 当前分类相关标签
4. 其他标签

如果某个标签同时属于“智能推荐”和“分类相关”，只展示一份，但在排序上应优先靠前。

### 7.3 无分类时的标签策略

如果用户尚未选择分类，则标签区只展示：

1. 已选标签
2. 智能推荐标签（若存在）
3. 默认常用标签 / 更多标签入口

此时不展示“当前分类相关标签”模块。

---

## 8. 数据与实现策略

### 8.1 本次推荐的实现策略

本次设计推荐使用：

- **前端配置驱动的标签分组与分类标签映射**
- 继续复用现有后端分类树和标签列表接口
- 继续复用现有 `/tag/recommend` 自动推荐能力

原因：

1. 当前后端 `tag/list` 只返回基本标签信息，未提供主题组字段
2. 当前需求重点是改善投稿选择体验，而不是重做标签数据模型
3. 前端配置方式改动更小，更适合渐进落地

### 8.2 前端配置内容

建议新增一份前端配置，用于维护：

1. 标签主题分组
2. 分类到标签的推荐映射
3. 常用标签顺序

这份配置不是“写死推荐结果”，而是：

- 提供默认组织结构
- 作为 UI 的优先展示依据
- 实际最终选择仍由用户决定

### 8.3 后端保持不变的部分

本次不要求修改：

- 分类树接口结构
- 标签列表接口结构
- 本地推荐标签接口结构

除非后续确认前端配置维护成本过高，再考虑把“标签主题组”迁到后端元数据。

推荐文案：

- `已选分类：科技 / 后端开发`
- `可继续细分到二级分类，也可直接使用一级分类`

### 9.2 标签区文案

推荐文案：

- `已选标签`
- `智能推荐`
- `当前分类相关标签`
- `更多标签`
- `搜索标签`

### 9.3 按钮文案

当前推荐按钮建议调整为：

- `智能推荐`

不再使用：

- `AI 推荐`

## 10. 边界情况

### 10.1 一级分类存在二级，但用户不想选二级

允许直接保留一级分类作为最终分类。

### 10.2 当前分类没有对应标签映射

不报错，直接隐藏“当前分类相关标签”模块，回退到：

- 智能推荐标签
- 更多标签

### 10.3 智能推荐为空

不报错，智能推荐区可以隐藏或显示空态提示。

### 10.4 标签很多

默认不全量展开，避免再次回到当前“满屏标签平铺”的问题。

---

## 11. 验收标准

本次改版完成后，应满足以下结果：

1. 用户可以清楚知道当前分类是否已完成选择
2. 用户在存在二级分类时，被清晰引导继续细分，但不会被强制卡死
3. 标签默认展示量明显下降，信息层次更清晰
4. 用户选定分类后，可以明显感知到标签列表已发生联动
5. 推荐标签文案不再使用“AI”表述
6. 投稿页整体视觉风格、布局骨架和 chip 风格保持一致

---

## 12. 测试建议

### 12.1 分类测试

1. 选择没有二级的一级分类，确认可直接提交
2. 选择有二级的一级分类，不选二级，确认允许提交
3. 选择有二级的一级分类后继续选择二级，确认路径显示正确

### 12.2 标签测试

1. 无分类时标签区显示默认结构
2. 选择分类后出现对应分类相关标签
3. 智能推荐标签可以正常加入 / 取消
4. 更多标签折叠与展开逻辑正常
5. 搜索标签仍可正常命中并选择

### 12.3 风格一致性测试

1. 分类与标签卡片视觉风格与现有投稿页一致
2. 不出现新的突兀视觉组件
3. 不破坏左右双栏布局和现有表单节奏

---

## 13. 最终结论

本次设计采用“**不改风格的渐进改版**”路线：

- 分类区：提升状态表达，允许一级兜底，二级优先但不强制
- 标签区：改成分层结构，重点引入“当前分类相关标签”层
- 推荐能力：保留现有自动推荐逻辑，但去掉 AI 叙事
- 技术策略：优先采用前端配置驱动，不强制改动后端标签数据模型

这套方案能在保留当前页面风格与结构稳定性的前提下，明显改善投稿时的选择效率与理解成本。
