# Category / Tag Taxonomy 重建设计

## 1. 背景

当前项目的 `category` 与 `tag` 体系已经出现明显的结构问题：

1. **分类维度混乱**：当前分类同时混入“内容领域”和“内容形式”两种概念，例如：
   - `生活`、`科技` 属于内容领域
   - `教程`、`解说`、`Vlog` 更偏内容形式或表达方式
2. **投稿选择困难**：当一个视频同时具备“领域 + 形式”双重特征时，用户会不知道该把它投到哪个分类。
3. **搜索 / 推荐特征不稳定**：像 `Java`、`前端`、`后端` 这类更适合作为标签的内容，被混进分类后，会削弱分类的稳定性，也影响推荐系统的可解释性。
4. **上传页联动基础不稳**：分类与标签映射只有在“分类结构本身干净”的前提下才有长期价值。

用户已经明确希望：

- **删除现有全部 `category` / `tag` 数据**
- 使用一套新的 taxonomy 体系完全重建
- 新体系同时服务：
  - 投稿体验
  - 搜索体验
  - 推荐 / 画像能力
- 同步更新前端标签分组与分类标签映射
- 这次视为**开发环境重置**，不要求兼容旧 ID，也不做旧数据迁移

---

## 2. 设计目标

### 2.1 目标

1. 让分类只承担“内容领域归属”的职责。
2. 让标签承担“主题、形式、技术、风格、方向”等细粒度特征表达。
3. 为投稿页提供更稳定的“分类 → 推荐标签”联动基础。
4. 为搜索与推荐提供更清晰的结构化特征。
5. 用一套可以长期维护的 taxonomy 替换当前混乱的初始化数据。

### 2.2 非目标

1. 本次不保留旧的 `category` / `tag` ID 兼容性。
2. 本次不尝试迁移线上或历史视频到新 taxonomy。
3. 本次不引入“系统隐藏标签”作为用户可选标签。
4. 本次不扩展数据库结构去支持 tag type / system tag / metadata tag。

---

## 3. 核心原则

## 3.1 分类只做“内容领域”

分类用于回答：

> 这个视频主要属于哪个内容分区？

因此分类只保留“领域/频道”语义，不再混入：

- 教程
- 解说
- 实战
- 评测
- 搞笑
- 干货

这些全部下放为标签。

## 3.2 标签做“内容特征”

标签用于回答：

> 这个视频具体讲了什么、以什么方式讲、有哪些风格和技术细节？

标签负责表达：

- 内容主题
- 表现形式
- 技术栈 / 工具
- 场景方向
- 平台内容风格

## 3.3 分类少而稳，标签多而细

- 分类数量必须控制，保持长期稳定
- 标签可以更丰富，但需要分组管理，避免投稿页再次陷入“全量平铺”

## 3.4 保留一个例外：Vlog

纯理论上，`Vlog` 更接近内容形式，应归入标签。

但基于平台用户心智与投稿体验，`Vlog` 作为一级分类非常强，保留它能显著降低投稿决策成本。因此本设计将 `Vlog` 作为唯一保留的“形式型一级分类特例”。

## 6.2 映射原则

1. 分类映射优先推荐“高相关、高频、易选”的标签。
2. 映射是“推荐”，不是“限制”，用户仍可跨组手动选择其他标签。
3. 标签映射优先服务上传页交互，同时也可作为推荐和搜索特征基础。

## 7. 本次不纳入 tag 表的内容

以下概念本身有价值，但不适合直接放进当前可选 tag 表：

- 短视频 / 中视频 / 长视频
- 原创 / 搬运
- 高清 / 4K
- 系列 / 单集

原因：

1. 它们更像系统属性或内容元数据，而不是用户主动选择的公开标签。
2. 若直接放进 tag 表，会污染投稿页的用户可选标签集合。
3. 更适合未来单独设计为：
   - 数据库字段
   - 枚举型内容属性
   - 隐藏推荐特征
   - tag type 扩展体系

---

## 8. 数据重建策略

## 8.1 清理策略

本次按“开发环境重置”执行，直接删除旧 taxonomy 相关数据。

建议清理对象：

- `video_tag`
- `video_tag_feature`
- `user_interest_tag`
- `tag`
- `category`

同时需要处理对分类/标签的引用关系，例如：

- `video.category_id`
- 以及视频已有标签关联

本设计建议在实现时采用一套明确的 reset 顺序，避免外键或逻辑脏数据残留。

## 8.2 不做旧数据迁移

由于用户已经明确接受“删掉旧 category/tag 全部重来”，因此本次不尝试做：

- 旧分类到新分类映射迁移
- 旧标签到新标签映射迁移
- 历史视频自动修复分类/标签

本次目标是得到一套**干净、一致、可长期维护**的新 taxonomy 基线。

---

## 9. 前端同步要求

## 9.1 需要同步更新的前端配置

重建 taxonomy 后，前端必须同步更新：

- `uploadTagConfig.js`
  - 标签主题分组
  - 默认常用标签
  - 分类 → 推荐标签映射

## 9.2 投稿页预期效果

重建完成后，上传页应具备更稳定的交互基础：

1. 用户先选择明确的领域分类
2. 系统根据分类展示相关标签
3. 再结合标题 / 简介返回智能推荐标签
4. 用户在“更多标签”里补充细节特征

这会让“分类选择”和“标签选择”在语义上更清晰，不再互相抢职责。

---

## 10. 验收标准

完成后应满足：

1. `category` 结构不再混入 `教程 / 解说 / 实战 / 评测 / 搞笑` 这类内容形式。
2. `科技` 等一级分类下不再拆到 `Java / 前端 / 后端` 这种技术粒度，而改由标签承担。
3. 投稿页选择分类时，分类更像“频道归属”；标签更像“特征补充”。
4. 前端标签分组和数据库 tag 初始化数据保持语义一致。
5. 分类 → 推荐标签映射可直接支撑上传页联动。
6. 新 taxonomy 可作为后续搜索、推荐、画像、造数的稳定基线。

---

## 11. 最终结论

本次 taxonomy 重建采用以下方向：

- **分类只管内容领域**
- **标签负责内容特征**
- **分类少而稳，标签多而细**
- **保留 `Vlog` 作为平台心智特例**
- **不兼容旧 ID，不迁移旧 taxonomy 数据**
- **后端初始化数据与前端标签配置同步重建**

这是最适合当前项目阶段、也最有利于上传页 / 搜索 / 推荐共同演进的一套方案。
