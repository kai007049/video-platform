# README 重构设计

## 1. 背景

用户希望重构根目录 `README.md`，用于项目正式发布到 GitHub。新的 README 需要同时满足两个目标：

1. 让访客快速理解项目的功能、价值和工程亮点。
2. 提供完整、清晰、可落地的启动步骤，帮助 fresh clone 用户把项目跑起来。

用户已确认：

- README 风格采用 **展示 + 落地并重** 的方案。
- 启动说明采用 **双模式并写**：最小可运行版 + 完整功能版。
- 亮点呈现采用 **业务能力与架构设计均衡** 的表达方式。

## 2. 目标

本次 README 重构目标如下：

1. 把根 README 从“简要说明”升级为“项目首页说明书”。
2. 让 GitHub 访客在短时间内看懂：项目做什么、有哪些功能、亮点是什么。
3. 让第一次 clone 仓库的人能按 README 的步骤完成环境准备、配置修改和项目启动。
4. 清楚说明哪些依赖是必需、哪些能力是可选，以及缺失时会退化什么功能。
5. 保持内容结构清晰，既适合展示，也适合排查启动问题。

## 3. 非目标

本次不做以下事情：

1. 不重写 `backend/README.md` 和 `frontend/README.md` 的全部内容。
2. 不新增部署到云服务器、Docker Compose 全家桶或生产环境运维文档。
3. 不承诺 README 中未实现的能力。
4. 不把 README 写成冗长的架构白皮书。
5. 不补截图、GIF、徽章等视觉素材，除非后续用户再提出。

## 4. 方案选择

采用 **方案 A：一个根 README，兼顾展示价值与可执行启动步骤**。

### 4.1 结构策略

新的根 README 分为三层：

#### 第一层：首页展示层
目标是让用户在 10 秒内判断这个仓库是否值得继续阅读。

包含：
- 项目名称与一句话简介
- 项目定位（学习 / 演示 / 工程实践型视频平台）
- 核心功能关键词
- 技术栈关键词
- 亮点摘要

#### 第二层：能力与亮点层
目标是兼顾“业务完整度”和“工程含金量”。

包含：
- 功能模块总览
- 架构亮点总览
- 搜索 / 推荐 / 缓存 / 异步 / 实时消息等专项亮点
- 项目结构说明

#### 第三层：落地启动层
目标是让 fresh clone 用户按照顺序完成环境准备和启动。

包含：
- 环境要求
- 依赖服务分级说明
- 最小启动模式
- 完整功能模式
- 配置修改说明
- 常见问题 / 启动失败排查

### 4.2 启动说明策略

README 中将同时提供两套启动路径：

#### 最小可运行模式
面向“先跑起来再说”的使用者。

特点：
- 优先跑通前后端和核心页面
- 说明必要依赖最小集合
- 明确哪些功能在此模式下会退化或不可用

#### 完整功能模式
面向“想体验完整链路”的使用者。

特点：
- 启用 MySQL、Redis、MinIO、Elasticsearch、RocketMQ、ffmpeg 等完整依赖
- 说明搜索、推荐、异步处理、对象存储、消息推送等能力的完整效果
- 提供更完整的配置准备说明

### 4.3 亮点表达策略

README 的亮点将均衡覆盖两类信息：

#### 业务亮点
- 视频上传、播放、互动、评论、弹幕、收藏、点赞
- 私信、通知、消息中心
- 搜索、推荐、投稿内容补全

#### 架构亮点
- 前后端分离
- Caffeine + Redis 两级缓存
- RocketMQ 异步解耦
- WebSocket 实时消息推送
- 多路召回、页面级重排、搜索索引同步
- 后端本地规则补全，不依赖独立 Python agent 服务

## 5. README 章节设计

新的根 `README.md` 计划采用以下章节顺序：

1. **项目简介**
   - 一句话说明项目是什么
   - 强调适合学习、演示、工程实践展示

2. **功能总览**
   - 以模块化方式列出主要业务能力

3. **项目亮点**
   - 分业务亮点与工程亮点两部分

4. **技术栈**
   - Backend / Frontend / Middleware 分类列出

5. **项目结构**
   - 根目录、backend、frontend、docs 的角色说明

6. **环境要求**
   - JDK、Node、MySQL、Redis 等版本建议

7. **依赖说明（必需 / 可选）**
   - 明确哪些依赖是最小启动必须，哪些是完整功能所需

8. **最小可运行模式启动步骤**
   - 按顺序写出：准备数据库 -> 修改配置 -> 启动后端 -> 启动前端
   - 明确退化能力

9. **完整功能模式启动步骤**
   - 在最小模式基础上补齐 MinIO / ES / RocketMQ / ffmpeg
   - 说明完整功能收益

10. **关键配置项说明**
    - `application.yaml` 中用户必须改的内容
    - 前端 API 代理 / baseURL 如有必要也说明

11. **常见问题**
    - 如启动失败、依赖未启动、搜索不可用、消息链路不完整、ffmpeg 路径问题等

12. **文档索引 / 参考文档**
    - 链接到 backend README、系统架构、缓存策略、推荐架构等

13. **项目说明 / 后续规划**
    - 说明项目定位与后续演进方向

## 6. 关键内容约束

### 6.1 启动步骤要求

启动步骤必须满足：

- 按执行顺序排列
- 具体到目录和命令
- 说明默认端口
- 说明需要修改哪些配置
- 说明启动后如何验证是否成功

### 6.2 依赖说明要求

依赖说明必须显式区分：

- **必需依赖**：项目最小可运行必须具备
- **增强依赖**：缺失时项目仍可启动，但部分功能退化

建议至少覆盖：
- MySQL
- Redis
- MinIO
- Elasticsearch
- RocketMQ
- ffmpeg

### 6.3 亮点描述要求

亮点必须基于当前仓库实际能力，避免夸大。表述风格应：

- 对外可展示
- 不营销过度
- 不写未完成能力
- 不把临时保留接口误写成完整特性

## 7. 风险与控制

### 风险

1. README 写得太展示化，会导致启动指引不够实用。
2. README 写得太操作手册化，会让 GitHub 首页失去吸引力。
3. 若依赖分级不清，用户仍可能误以为所有中间件都是强依赖。
4. 若功能描述与当前实现不一致，会降低项目可信度。

### 控制措施

1. 采用“展示层 + 能力层 + 启动层”三层结构。
2. 明确写出“最小模式”和“完整模式”。
3. 对每种可选依赖都说明其作用和缺失后的影响。
4. 只保留当前仓库中已经具备的功能与亮点描述。

## 8. 验收标准

满足以下条件即可视为本次 README 重构完成：

1. 根 README 能清楚说明项目定位、主要功能和工程亮点。
2. 根 README 包含最小可运行模式和完整功能模式两套启动说明。
3. README 明确区分必需依赖和可选依赖。
4. README 写清楚主要配置修改点和默认启动端口。
5. README 增加常见问题 / 启动排查信息。
6. README 首页既适合 GitHub 展示，也足够支撑首次启动。

## 9. 结论

本次采用一个主 README 统一承载“项目展示”和“启动落地”两类职责。通过三层结构和双模式启动说明，兼顾 GitHub 首页观感、工程亮点表达以及 fresh clone 用户的上手体验。
