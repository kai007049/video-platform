# Video Platform 系统架构说明

## 1. 项目定位

本项目是一个面向视频内容平台场景的全栈系统，覆盖用户、投稿、播放、互动、搜索、推荐、私信、通知等核心链路。项目采用前后端分离架构，后端以 Spring Boot 为核心，前端以 Vue 3 为核心。

这个项目更适合作为学习、展示和面试讲解的工程项目，而不是直接面向生产环境的商业系统。它的重点不是堆功能，而是把视频平台里常见的缓存、搜索、推荐、异步解耦、对象存储、实时消息等工程问题做成一套可以运行和解释的实现。

## 2. 总体架构

```text
用户浏览器
   |
   v
Vue 3 前端
   |
   v
Spring Boot 后端
   |
   |-- MySQL：业务主数据
   |-- Redis：缓存、状态数据、热榜数据
   |-- Elasticsearch：搜索索引
   |-- MinIO：视频、封面、头像、消息图片等对象存储
   |-- RocketMQ：异步任务与系统解耦
```

核心设计思路：

- MySQL 作为最终一致的数据源
- Redis 承接高频缓存、计数、热榜和状态类数据
- Elasticsearch 承接搜索能力
- MinIO 承接媒体文件和图片文件
- RocketMQ 承接视频后处理、搜索同步、通知推送、资源清理等异步链路
- WebSocket 承接实时消息推送

## 3. 后端分层

```text
controller  ->  service  ->  mapper  ->  MySQL
                  |
                  |-- Redis / Caffeine
                  |-- Elasticsearch
                  |-- MinIO
                  |-- RocketMQ
                  |-- WebSocket
```

主要包职责：

- `controller`：接收 HTTP 请求，处理参数和统一响应
- `service`：承载业务逻辑，协调数据库、缓存、搜索、对象存储和消息队列
- `mapper`：基于 MyBatis-Plus 访问 MySQL
- `entity`：数据库实体
- `model` / `vo`：接口返回对象和页面展示模型
- `config`：Spring、Redis、跨域、拦截器、OpenAPI 等配置
- `mq`：消息生产与消费逻辑
- `utils`：JWT、上下文、MinIO、通用工具能力
- `seed`：本地演示数据生成与推荐链路验证辅助

## 4. 核心业务域

### 用户域

- 注册、登录、JWT 鉴权
- 用户主页、创作者信息
- 头像上传与访问

### 视频域

- 视频上传、删除、详情、播放、观看历史
- 视频列表查询与聚合展示
- 封面、预览图、播放地址等媒体信息管理

### 互动域

- 点赞、收藏、评论、弹幕、关注
- 列表和详情页批量聚合互动状态，避免逐条查询

### 消息域

- 私信会话
- 通知与系统消息
- 消息中心未读数维护
- WebSocket 实时推送

### 搜索域

- Elasticsearch 关键词搜索
- 搜索历史
- 热搜关键词
- 搜索索引同步与全量重建

### 推荐域

- backend-only 推荐系统
- 多路召回、线性打分、页面级重排
- 曝光日志记录
- seed 数据生成辅助验证推荐结果

## 5. 关键链路

### 视频上传链路

```text
前端上传视频 / 封面
   ->
后端接收文件
   ->
上传到 MinIO
   ->
写入 MySQL 视频记录
   ->
发送 RocketMQ 消息
   ->
异步处理封面、搜索索引、通知等后置任务
   ->
失效相关缓存
```

### 视频详情读取链路

```text
前端请求视频详情
   ->
Caffeine 热点本地缓存
   ->
Redis 视频详情缓存
   ->
MySQL 回源与业务聚合
   ->
返回完整 VideoVO
```

### 搜索链路

```text
前端发起搜索
   ->
Elasticsearch 检索标题 / 描述
   ->
补查 MySQL 业务数据
   ->
组装搜索结果
```

### 推荐链路

```text
用户进入推荐页
   ->
热门 / 新鲜 / 标签 / 分类 / 作者 / 运营推荐多路召回
   ->
特征打分
   ->
页面级重排
   ->
记录曝光日志
   ->
返回推荐结果
```

## 6. 工程亮点

### 缓存体系

- 视频详情使用 Redis + Caffeine 两级缓存
- 高频统计值和相对稳定的详情信息分层维护
- 热榜数据使用 Redis ZSet 维护
- 分类、标签、未读数、搜索历史等高频读数据接入 Redis

### 搜索与推荐

- 搜索链路基于 Elasticsearch 构建
- 推荐系统采用多路召回、可解释打分、页面级重排
- 曝光日志记录推荐来源和策略版本，便于分析推荐结果

### 异步解耦

- RocketMQ 用于搜索同步、通知推送、视频后处理、资源清理等链路
- 主流程优先保证核心状态落库
- 后置任务通过消息机制异步执行，降低主链路耗时和模块耦合

### 实时消息

- 私信、通知、系统消息形成完整消息中心
- WebSocket 用于在线用户实时推送
- 未读数通过缓存提升读取效率

## 7. 当前边界

当前版本已经完成核心业务闭环和主要工程链路，但仍保留继续增强空间：

- 推荐系统仍以可解释规则和线性打分为主
- 搜索排序仍可继续引入更多业务特征
- 多实例部署下的本地缓存一致性仍需进一步治理
- 文件处理链路可以继续演进到分片上传、转码、HLS 播放
- 部署层面还可以补充完整 Dockerfile、CI/CD 和监控体系

## 8. 面试讲解建议

讲这个项目时，可以按下面顺序展开：

1. 先讲整体业务闭环：用户、视频、互动、搜索、推荐、消息
2. 再讲核心工程问题：缓存、异步、搜索、推荐、对象存储
3. 重点讲取舍：为什么用 Redis、为什么引入 MQ、为什么推荐先做可解释模型
4. 最后讲演进：多实例缓存一致性、推荐反馈闭环、搜索排序优化、视频转码和部署工程化

## 9. 总结

这个系统的核心价值在于：它不仅完成了视频平台的主业务链路，也围绕缓存、搜索、推荐、异步解耦、对象存储、实时消息这些工程主题做出了可以运行、可以解释、可以继续演进的实现。
