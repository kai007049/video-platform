# 缓存策略说明

## 1. 设计目标

缓存体系的目标是降低数据库访问压力、提升高频接口响应速度，并为后续多实例部署和高并发场景预留扩展空间。

本项目的缓存不是简单地“给接口加 Redis”，而是围绕视频平台的访问特点做了分层：

- 视频详情内容缓存
- 高频统计值缓存
- 热点本地缓存
- 元数据缓存
- 搜索与消息辅助缓存
- 热门榜缓存

## 2. 缓存总体结构

```text
请求视频详情
   ->
Caffeine 热点本地缓存
   ->
Redis 统一缓存层
   ->
MySQL 回源
```

其中：

- Caffeine：进程内缓存，只承接热点视频详情
- Redis：跨实例共享缓存，承接详情、统计、元数据、热榜等数据
- MySQL：最终真实数据来源

这种设计兼顾了访问速度和分布式扩展能力。

## 3. 视频详情缓存

视频详情被拆成相对稳定的基础信息和高频变化的统计信息。

### 基础信息

典型 key：

```text
video:base:v2:{id}
```

适合放入基础信息的字段：

- 标题
- 描述
- 作者信息
- 封面 / 预览图 / 视频地址
- 时长
- 分类
- 推荐位
- 创建时间

这些字段变化频率较低，适合作为一个整体缓存。

### 统计信息

典型 key：

```text
video:stat:{id}
```

使用 Redis Hash 存储：

- `play`
- `like`
- `save`

这些字段变化频率高，适合使用 `HINCRBY` 做字段级原子更新。这样播放、点赞、收藏等操作不需要频繁失效整个详情缓存。

## 4. 热点本地缓存

视频详情页存在明显热点，部分视频会在短时间内被大量访问。为了降低 Redis 和数据库压力，项目使用 Caffeine 做热点本地缓存。

本地缓存不缓存所有视频，而是收敛到热视频：

- 热门榜由 Redis ZSet 维护
- 热榜 Top N 的视频允许进入 Caffeine
- 长尾视频仍主要走 Redis

这样可以避免长尾数据挤占本地缓存空间。

## 5. 元数据缓存

分类、标签等数据读取频率高、修改频率低，适合缓存：

- 分类树
- 分类列表
- 标签列表

这类缓存能减少页面初始化、投稿页、搜索筛选等场景的重复数据库查询。

## 6. 搜索和消息辅助缓存

项目中还缓存了部分辅助数据：

- 搜索历史
- 热搜关键词
- 私信未读数
- 通知未读数

这些数据通常读取频繁、更新粒度小，使用 Redis 可以明显降低数据库压力。

## 7. 热门榜缓存

热门榜使用 Redis ZSet 维护：

```text
video:hot:24h
```

播放、点赞、收藏等行为发生后会影响热视频分数。热门榜同时服务：

- `/video/hot`
- 推荐系统中的热门召回通道
- 热点本地缓存的视频范围判断

这让热榜不只是一个页面功能，也成为推荐和缓存体系的基础设施。

## 8. 缓存失效策略

项目主要采用“更新数据库后删除缓存”的策略。

典型场景：

- 视频上传后刷新相关列表
- 视频删除后失效详情和列表缓存
- 封面处理完成后失效视频详情
- MQ 消费完成后统一触发缓存失效

选择删除缓存而不是直接更新缓存，是因为视频展示对象往往是聚合结果，直接更新容易漏字段、漏场景。删除后下次访问回源重建，整体一致性更容易控制。

## 9. 为什么不直接缓存完整列表结果

推荐、搜索这类接口如果直接缓存完整 `VideoVO` 列表，会带来较高失效成本：

- 视频标题、封面、统计值变化后需要同时维护多份结果缓存
- 用户态字段如点赞、收藏、观看进度不适合进入公共缓存
- 完整对象体积更大，缓存空间和网络开销更高

更合适的方向是：

- 推荐结果缓存 `videoIds`
- 搜索结果缓存 `ids + total`
- 命中结果缓存后继续复用视频详情缓存和统一组装逻辑

这能让结果层缓存和内容层缓存保持解耦。

## 10. 当前边界

当前缓存体系已经覆盖视频详情、热点、本地缓存、元数据、搜索辅助、未读数和热榜等核心场景。仍可继续增强的方向包括：

- 推荐结果缓存
- 搜索结果缓存
- 热点 key 重建保护
- 多实例本地缓存失效广播
- Caffeine / Redis 命中率监控
- DB fallback 次数统计

## 11. 面试讲解建议

可以重点讲这些问题：

- 为什么视频详情适合做缓存
- 为什么统计值要和基础信息拆开
- 为什么本地缓存只缓存热点视频
- 为什么推荐和搜索更适合缓存 ID 列表而不是完整对象
- 多实例部署后本地缓存会带来什么一致性问题
- 缓存删除、TTL、主动失效各自解决什么问题

## 12. 总结

本项目的缓存策略核心是：用 Redis 承接跨实例共享缓存，用 Caffeine 承接热点本地加速，用分层 key 设计降低高频统计更新对详情缓存的影响，并为后续推荐、搜索结果缓存留下清晰扩展空间。
