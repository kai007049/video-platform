# 视频详情缓存分层说明

## 1. 背景

视频详情页是视频平台中访问频率最高的页面之一。一个详情页通常包含两类数据：

- 基础信息：标题、简介、封面、作者、分类、视频地址等
- 统计信息：播放数、点赞数、收藏数等

这两类数据的变化频率不同。如果全部作为一个大对象缓存，高频统计字段变化时就会影响低频基础信息，导致缓存频繁失效。

## 2. 分层目标

视频详情缓存分层的目标是：

1. 将低频基础信息和高频统计信息拆开维护
2. 让播放、点赞、收藏等高频操作可以做字段级原子更新
3. 保持详情读取路径简单
4. 降低缓存失效和重建成本
5. 为热点本地缓存、多实例扩展打基础

## 3. Key 设计

### 基础信息

```text
video:base:v2:{videoId}
```

用于缓存相对稳定的视频展示信息。

### 统计信息

```text
video:stat:{videoId}
```

使用 Redis Hash 存储：

```text
play
like
save
```

字段级更新示例：

```text
HINCRBY video:stat:{videoId} play 1
HINCRBY video:stat:{videoId} like 1
HINCRBY video:stat:{videoId} save 1
```

批量读取示例：

```text
HMGET video:stat:{videoId} play like save
```

## 4. 读取流程

```text
请求视频详情
   ->
查询 Caffeine 热点本地缓存
   ->
未命中则读取 Redis base
   ->
读取 Redis stat
   ->
必要时回源 MySQL 并重建缓存
   ->
组装 VideoVO
   ->
返回前端
```

读路径仍然保持清晰：基础信息和统计信息分开缓存，最终由后端组装成页面需要的展示对象。

## 5. 写入流程

### 播放

```text
记录播放行为
   ->
更新 MySQL / Redis 统计
   ->
更新热榜分数
```

### 点赞 / 收藏

```text
写入互动关系
   ->
更新统计字段
   ->
必要时失效相关详情缓存
```

### 视频基础信息变化

```text
更新标题 / 封面 / 分类等基础字段
   ->
删除 video:base:v2:{videoId}
   ->
下次访问时回源重建
```

## 6. 为什么 stats 使用 Hash

统计字段可以有多种存法：

- 每个统计字段一个独立 key
- 整个统计对象作为 JSON 缓存
- 一个 Redis Hash 存多个字段

当前选择 Redis Hash，原因是：

- 一个 key 表达一个视频的统计对象，语义清晰
- 支持字段级原子增减
- 多个统计字段可以一次性读取
- TTL 和删除逻辑更容易统一
- 与播放、点赞、收藏等高频更新场景匹配

## 7. 评论数的处理

评论数和播放、点赞、收藏的统计口径不同。评论涉及新增、删除、回复等场景，且通常需要以评论表聚合结果为准。

因此当前设计不直接把评论数并入 `video:stat:{videoId}`，避免 Redis 统计值和数据库真实评论数据产生口径分裂。

## 8. 与热点本地缓存的关系

视频详情缓存分层后，本地缓存更适合承接完整详情展示对象，但只服务热点视频：

- Redis 负责统一缓存层
- Caffeine 负责热点加速层
- 热门榜用于判断哪些视频可以进入本地缓存

这样既能提升热视频访问速度，也能避免所有视频都进入进程内缓存。

## 9. 与推荐 / 热门的关系

视频统计字段不仅服务详情页，也会影响：

- 热门榜排序
- 推荐系统热门召回
- 推荐系统中的热度特征

因此统计值缓存需要保持清晰的更新规则，避免推荐和热门结果使用过期或不一致的数据。

## 10. 后续扩展

后续可以继续增强：

- 热点 key 重建保护
- Caffeine 命中率监控
- Redis 命中率监控
- 多实例本地缓存失效广播
- 统计值定期校准

## 11. 总结

视频详情缓存分层的核心思路是：低频基础信息整体缓存，高频统计信息独立缓存并支持原子更新。这样可以减少缓存失效范围，提升热点详情页性能，也为后续推荐、热门和多实例缓存治理提供更稳定的基础。
