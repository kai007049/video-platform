# Video-Platform 缓存策略说明

## 1. 缓存设计目标

本项目引入缓存的主要目标有三个：

- 降低高频热点数据对数据库的访问压力
- 提升视频详情、视频列表等核心接口的响应速度
- 为后续高并发场景下的扩展预留能力

当前缓存体系主要围绕视频相关数据展开，核心由本地缓存和 Redis 组成。

## 2. 当前缓存架构

项目当前采用的是“两级缓存 + 失效控制”的思路：

```text
请求
  ->
本地缓存（Caffeine，仅热点视频详情）
  ->
Redis
  ->
数据库 / 业务组装
```

其中：

- Caffeine：作为进程内本地缓存，当前已收敛为**只缓存热榜 Top N 的热点视频详情**，减少 Redis 网络开销
- Redis：作为分布式缓存，承接所有视频详情的统一主缓存层
- 数据库：作为最终真实数据来源

## 3. 为什么要使用两级缓存

### 3.1 本地缓存的价值

本地缓存访问速度非常快，适合承接真正热点视频详情的高频读取。对于单机开发和单实例部署场景，命中本地缓存可以显著减少 Redis 和数据库压力；对于后续多实例场景，本地缓存也更适合作为热点加速层，而不是承载所有长尾详情数据。

### 3.2 Redis 的价值

Redis 适合承载跨实例共享的数据，当系统部署多个后端实例时，本地缓存无法天然同步，这时 Redis 可以承担统一缓存层的职责。

### 3.3 两级缓存组合的意义

两级缓存的价值在于：

- 先尽量命中本地热点缓存，提高读取速度
- 本地未命中时再访问 Redis
- Redis 未命中时再回源数据库

这样既保留了高性能，又兼顾了分布式场景下的数据共享能力。

## 4. 当前主要缓存对象

当前缓存重点聚焦在视频详情及其附属展示数据上，例如：

- 视频基础信息
- 作者信息
- 点赞、收藏、评论等展示相关字段
- 播放统计等附属信息

之所以优先缓存视频相关数据，是因为视频详情页和视频列表页通常访问频率最高，也是最容易成为热点的部分。

## 5. 视频详情缓存流程

以视频详情查询为例，当前缓存读取流程可以概括为：

```text
查询视频详情
  ->
先查 Caffeine（仅热点视频可命中）
  ->
未命中则查 Redis
  ->
仍未命中则查询数据库并组装 VO
  ->
回写 Redis
  ->
若该视频属于热榜 Top N，则回写 Caffeine
  ->
返回结果
```

当前项目里，热门视频的判定直接复用 Redis 热榜 `video:hot:24h`，并以热榜 Top 500 作为允许进入本地缓存的标准。

这类设计的核心目的是：
- 让 Redis 继续承担统一详情缓存层
- 让本地缓存只服务真正热点视频
- 降低长尾视频对本地缓存空间的污染

## 6. 为什么要统一缓存入口

在重构过程中，视频缓存相关逻辑被统一收口到 `VideoCacheService` 中，核心原因是：

- 避免不同 Service 各自维护一套缓存规则
- 统一缓存 key、过期时间、失效方式
- 降低缓存一致性问题的复杂度
- 便于后续扩展更多缓存对象

如果缓存逻辑分散在多个类里，后续在新增字段、修改失效策略、排查缓存异常时都会很痛苦。

## 7. 缓存失效策略

### 7.1 为什么不能只依赖 TTL

如果只依赖过期时间，数据更新后缓存可能长时间不刷新，导致页面看到旧数据。因此对于视频相关的变更操作，需要主动失效。

### 7.2 当前主要失效方式

项目当前更偏向“更新数据库后主动删除缓存”的策略。

典型场景包括：

- 视频上传后，相关列表缓存需要刷新
- 视频删除后，详情缓存和相关列表缓存需要失效
- 视频封面处理完成后，视频详情缓存需要失效
- MQ 消费完成后，统一调用缓存失效入口

这种做法的优点是实现相对简单，而且对于读多写少场景比较适合。

### 7.3 为什么优先选择删除缓存而不是直接更新缓存

直接更新缓存虽然看起来更即时，但会带来两个问题：

- 写路径变复杂，容易漏改
- 缓存对象往往是聚合后的 VO，不一定适合逐字段修改

因此当前阶段优先采用“删缓存，下次访问再重建”的策略，更容易保证一致性。

## 8. 缓存与 N+1 查询优化的关系

缓存解决的是“热点数据重复读取”的问题，而批量查询解决的是“列表聚合时数据库查询次数爆炸”的问题。这两者并不是替代关系，而是互补关系。

例如视频列表页的优化过程是：

- 先查询视频主列表
- 批量加载作者、评论数、点赞状态、收藏状态、观看进度等附属数据
- 使用 `Map` 进行内存回填

这样做解决了 N+1 查询问题，避免在遍历视频列表时反复查询数据库。

而缓存则进一步解决：

- 相同视频详情被重复访问时，避免每次都回源数据库

可以理解为：

- 批量查询优化的是“单次请求内部”
- 缓存优化的是“多次请求之间”

## 9. 当前缓存方案的优点

### 9.1 实现成本适中

相比一开始就引入复杂的缓存同步和事件驱动机制，当前方案更容易实现和维护，适合项目现阶段规模。

### 9.2 对热点详情页收益明显

视频详情页天然适合缓存，而当前策略进一步把本地 Caffeine 收敛到热榜 Top 500 视频详情，使本地缓存更集中服务真正高频数据。这样既保留了热点详情页的高命中收益，也避免了大量长尾视频挤占本地缓存空间。

### 9.3 支持后续平滑升级

当前已经具备统一缓存入口，后面如果需要扩展：

- 批量 Redis 读取
- 热点 key 保护
- 统计值独立缓存
- 分布式缓存更新

改造成本会更低。

## 10. 当前方案的风险点

### 10.1 本地缓存一致性问题

当系统扩展为多实例部署时，不同实例中的 Caffeine 数据天然不会自动同步，因此会出现短时间的数据不一致问题。

### 10.2 聚合缓存更新复杂

缓存对象往往是聚合后的结果，不只是数据库单表字段，因此更新时通常不适合局部修改，只能删后重建。

### 10.3 高并发下可能出现缓存击穿

如果某个热点视频缓存刚好失效，短时间内可能有大量请求同时回源数据库。当前方案虽然已经具备一定保护思路，但后续仍可以继续加强。

## 11. 后续可优化方向

### 11.1 热点统计值拆分缓存

播放数、点赞数、评论数、收藏数等统计值可以考虑单独缓存，而不是全部揉进完整详情缓存中，这样更利于高频更新。

### 11.2 Redis 批量化读取

列表页如果后续接入更多缓存，可以进一步考虑 Redis 批量读取能力，减少网络往返。

### 11.3 增加缓存击穿保护

后续可以进一步加强：

- 分布式锁
- 随机过期时间
- 热点 key 永不过期 + 异步刷新

### 11.4 构建更明确的缓存规范

建议后续形成统一规范，至少明确：

- 什么数据值得缓存
- 缓存 key 如何命名
- 谁负责失效
- 更新后采用删缓存还是写缓存
- 过期时间如何设置

## 12. 面试时可重点讲的缓存点

可以重点从以下角度展开：

- 为什么视频详情适合做两级缓存
- 为什么当前阶段选择“删缓存而不是写缓存”
- 为什么缓存入口要统一收口
- 为什么缓存和 N+1 优化是两类不同问题
- 如果系统扩展为多实例部署，本地缓存会带来什么一致性问题

## 13. 总结

本项目当前缓存策略的核心思想是：围绕高频热点查询场景，优先用两级缓存降低数据库压力，并通过统一缓存入口和主动失效机制控制复杂度。在项目现阶段，这是一种兼顾性能、可维护性和实现成本的方案；后续随着数据规模和并发量增长，还可以继续向统计值拆分缓存、分布式一致性优化和缓存规范化方向演进。
