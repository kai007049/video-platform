# Video-Platform 缓存策略说明

## 1. 缓存设计目标

本项目引入缓存的主要目标有三个：

- 降低高频热点数据对数据库与搜索链路的访问压力
- 提升视频详情、首页推荐、搜索等核心接口的响应速度
- 为后续高并发和多实例场景下的扩展预留能力

当前缓存体系已经不再只是“给某几个接口加 Redis”，而是逐步形成了：

- 内容层缓存
- 元数据缓存
- 辅助缓存
- 结果层缓存（已设计 / 部分推进中）

---

## 2. 当前缓存架构

项目当前缓存体系可以分成两层理解：

### 2.1 内容层缓存

```text
请求视频详情
  ->
本地缓存（Caffeine，仅热点视频详情）
  ->
Redis
  ->
数据库 / 业务组装
```

其中：

- Caffeine：作为进程内本地缓存，当前已收敛为**只缓存热榜 Top 500 的热点视频详情**
- Redis：作为统一主缓存层，承接所有视频详情内容与统计层数据
- 数据库：作为最终真实数据来源

### 2.2 结果层缓存（当前状态：已设计，部分正在推进）

结果层缓存不直接缓存完整 `VideoVO`，而是缓存：

- 推荐结果窗口 `videoIds`
- 搜索结果 `ids + total`

这样可以继续复用内容层缓存与 `VideoViewAssembler` 的组装逻辑，避免缓存完整聚合对象带来的失效复杂度。

---

## 3. 当前主要缓存对象

### 3.1 视频详情内容层
当前视频详情缓存已演进为：

```text
video:base:v2:{id}
video:stat:{id}
```

其中：

#### `video:base:v2:{id}`
负责低频变化、稳定字段，例如：
- 标题
- 描述
- 作者信息
- 封面 / 预览图 / 视频地址
- 时长
- 推荐位
- 分类
- 评论数与统计基线

#### `video:stat:{id}`
使用 Redis Hash 存储高频变化统计字段：
- `play`
- `like`
- `save`

并通过字段级原子操作维护，例如：
- `HINCRBY`
- `HMGET`

### 3.2 元数据缓存
已经缓存：
- 分类树
- 分类列表
- 标签列表

这类对象低频变更、高频读取，非常适合作为 Redis 元数据缓存。

### 3.3 搜索辅助缓存
已经缓存：
- 搜索历史
- 热搜关键词聚合结果

### 3.4 未读数缓存
已经缓存：
- 私信未读数
- 通知未读数

### 3.5 热门榜缓存
已经使用 Redis ZSet 维护全局热视频榜：
- `video:hot:24h`

热门榜会在播放、点赞、收藏等行为发生后更新，并被：
- `/video/hot`
- 推荐系统的热门召回通道
共同复用。

---

## 4. 为什么要使用两级缓存

### 4.1 本地缓存的价值
本地缓存访问速度非常快，适合承接真正热点视频详情的高频读取。对于单机开发和单实例部署场景，命中本地缓存可以显著减少 Redis 和数据库压力；对于后续多实例场景，本地缓存也更适合作为热点加速层，而不是承载所有长尾详情数据。

### 4.2 Redis 的价值
Redis 适合承载跨实例共享的数据，当系统部署多个后端实例时，本地缓存无法天然同步，这时 Redis 可以承担统一缓存层的职责。

### 4.3 两级缓存组合的意义
两级缓存的价值在于：

- 先尽量命中本地热点缓存，提高读取速度
- 本地未命中时再访问 Redis
- Redis 未命中时再回源数据库

这样既保留了高性能，又兼顾了分布式场景下的数据共享能力。

---

## 5. 视频详情缓存流程

当前视频详情读取流程可以概括为：

```text
查询视频详情
  ->
先查 Caffeine（仅热点视频可命中）
  ->
未命中则查 Redis base
  ->
读取 Redis stat
  ->
合并为完整 VideoVO
  ->
若该视频属于热榜 Top N，则回填 Caffeine
  ->
返回结果
```

当前项目里，热门视频的判定直接复用 Redis 热榜 `video:hot:24h`，并以热榜 Top 500 作为允许进入本地缓存的标准。

这类设计的核心目的是：
- 让 Redis 继续承担统一详情缓存层
- 让本地缓存只服务真正热点视频
- 让高频统计字段独立出来，避免整对象频繁失效

---

## 6. 为什么要统一缓存入口

视频缓存相关逻辑当前统一收口在 `VideoCacheService` 中，核心原因是：

- 避免不同 Service 各自维护一套缓存规则
- 统一缓存 key、过期时间、失效方式
- 降低缓存一致性问题复杂度
- 便于后续扩展更多缓存对象

如果缓存逻辑分散在多个类里，后续在新增字段、修改失效策略、排查缓存异常时都会很痛苦。

## 7. 缓存失效策略

### 7.1 为什么不能只依赖 TTL
如果只依赖过期时间，数据更新后缓存可能长时间不刷新，导致页面看到旧数据。因此对于视频相关的变更操作，需要主动失效。

### 7.2 当前主要失效方式
项目当前更偏向“更新数据库后主动删除缓存”的策略。

典型场景包括：
- 视频上传后，相关列表缓存需要刷新
- 视频删除后，详情缓存和相关列表缓存需要失效
- 视频封面处理完成后，视频详情缓存需要失效
- MQ 消费完成后，统一调用缓存失效入口

### 7.3 为什么优先选择删除缓存而不是直接更新缓存
直接更新缓存虽然看起来更即时，但会带来两个问题：

- 写路径变复杂，容易漏改
- 缓存对象往往是聚合结果，不一定适合逐字段修改

因此当前阶段优先采用“删缓存，下次访问再重建”的策略，更容易保证一致性。

---

## 8. 当前缓存体系的价值

### 8.1 对热点详情页收益明显
视频详情页天然适合缓存，而当前策略进一步把本地 Caffeine 收敛到热榜 Top 500 视频详情，使本地缓存更集中服务真正高频数据。这样既保留了热点详情页的高命中收益，也避免了大量长尾视频挤占本地缓存空间。

### 8.2 对统计写入场景更合理
将 `play / like / save` 拆到 `video:stat:{id}` 后，播放、点赞、收藏等高频写入不再拖着整个详情对象缓存一起失效，缓存粒度更合理。

### 8.3 为结果层缓存打基础
有了稳定的内容层缓存之后，后续推荐结果缓存、搜索结果缓存就可以只缓存 `ids` 或 `ids + total`，而不必缓存完整 `VideoVO`，这让体系更清晰。

### 8.4 为推荐 / 热门验证提供基础设施
当前 seed generator 和 behavior simulator 都会复用：
- 视频统计字段
- Redis 热门榜
- 推荐 / 热门接口

这意味着缓存层不仅服务真实业务流，也已经成为推荐验证工具链的一部分。

---

## 9. 当前方案的风险点

### 9.1 本地缓存一致性问题
当系统扩展为多实例部署时，不同实例中的 Caffeine 数据天然不会自动同步，因此会出现短时间的数据不一致问题。

### 9.2 结果层缓存仍在推进中
当前推荐结果缓存、搜索结果缓存正在逐步设计与实现，尚未完全成为成熟稳定的统一方案，因此 README 和架构文档中更适合定位成“已设计 / 正在推进”。

### 9.3 高并发下可能出现缓存击穿
无论是热点详情缓存还是未来的推荐 / 搜索结果缓存，只要 key 足够热，就可能出现集中失效带来的击穿问题，因此后续仍需要进一步加强重建保护。

---

## 10. 后续可优化方向

### 10.1 结果层缓存稳定化
- 推荐结果窗口缓存
- 搜索结果缓存
- 推荐 / 搜索热点 key 的重建保护

### 10.2 热点集合本地快照
目前热点视频判定仍会访问 Redis 热榜 rank，后续可以给热点集合增加短 TTL 本地快照，减少每次详情请求的 Redis 额外访问。

### 10.3 命中率与回源监控
建议后续补齐：
- Caffeine 命中率
- Redis 命中率
- DB fallback 次数
- 推荐结果缓存命中率
- 搜索结果缓存命中率

### 10.4 多实例下一致性增强
后续可继续考虑：
- Redis pub/sub 广播失效
- MQ 广播本地缓存失效
- 热点 key 异步刷新

---

## 11. 面试时可重点讲的缓存点

可以重点从以下角度展开：

- 为什么视频详情适合做两级缓存
- 为什么本地缓存应该收敛为热点缓存
- 为什么统计值适合拆成 `base + stats`
- 为什么结果层缓存不该直接缓存完整 `VideoVO`
- 为什么缓存和 N+1 查询优化是两类不同问题
- 如果系统扩展为多实例部署，本地缓存会带来什么一致性问题
- 为什么推荐 / 热门验证也要考虑缓存层影响

---

## 12. 总结

本项目当前缓存策略的核心思想可以总结为：

> 围绕高频热点读取场景，优先用内容层两级缓存降低数据库压力；围绕高计算链路，逐步引入结果层缓存；同时通过统一缓存入口和主动失效机制控制复杂度。

在项目现阶段，这是一种兼顾性能、可维护性和实现成本的方案；后续随着数据规模和并发量增长，还可以继续向结果层缓存、命中率监控、多实例一致性优化方向演进。
