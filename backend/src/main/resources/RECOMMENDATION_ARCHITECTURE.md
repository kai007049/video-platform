# 推荐系统架构说明

## 1. 目标

当前版本的推荐系统完全基于 backend 实现，不依赖独立 agent-service。

这样做的目标有四点：

1. 先把推荐链路做稳定，保证系统在纯 Java 后端下可以独立运行。
2. 充分利用用户行为、内容标签、分类、作者偏好等一方数据。
3. 保持推荐策略可解释、可调试、可持续演进。
4. 通过 seed generator 和 behavior simulator 让推荐结果具备可重复验证能力。

---

## 2. 总体设计

推荐系统按四层组织：

### 2.1 召回层（Recall）
- 热门召回
- 新鲜召回
- 标签兴趣召回
- 分类偏好召回
- 作者偏好召回
- 运营推荐召回

### 2.2 特征构建层（Feature Build）
- 热度分
- 新鲜度分
- 标签兴趣匹配分
- 分类偏好匹配分
- 作者偏好匹配分
- 运营推荐加分
- 最近看过惩罚

### 2.3 打分层（Ranking）
采用线性公式融合候选视频特征分数，便于调试与调权。

### 2.4 重排层（Rerank）
- 控制同作者数量
- 控制同分类数量
- 限制同分类连续出现
- 降低最近看过内容的重复曝光

---

## 3. 各召回通道说明

### 3.1 热门召回
数据来源：Redis 热榜 ZSet。

### 3.2 新鲜召回
数据来源：按 `create_time` 倒序查询最近发布的视频。

### 3.3 标签兴趣召回
数据来源：
- 用户兴趣画像 `user_interest_tag`
- 视频标签特征 `video_tag_feature`

### 3.4 分类偏好召回
数据来源：
- 最近看过的视频分类
- 点赞/收藏过的视频分类

### 3.5 作者偏好召回
数据来源：
- 最近高频互动作者
- 已关注作者

### 3.6 运营推荐召回
数据来源：`video.isRecommended`

---

## 4. 打分思路

```text
finalScore =
  hotWeight * hotScore
+ freshWeight * freshScore * freshnessDecay
+ tagWeight * interestTagScore
+ categoryWeight * categoryAffinityScore
+ authorWeight * authorAffinityScore
+ editorialWeight * editorialBoost
- recentWatchPenalty
```

其中：
- `freshnessDecay` 用于控制新鲜度衰减
- `recentWatchPenalty` 用于减少短时间重复推荐

---

## 5. 重排策略

在排序后增加页面级重排：
- 单页同作者数量限制
- 单页同分类数量限制
- 同分类连续出现数量限制

目标是保证推荐流既相关又丰富。

---

## 6. 冷启动策略

### 6.1 新用户冷启动
主要依赖：
- 热门召回
- 新鲜召回
- 运营推荐召回

### 6.2 新视频冷启动
主要依赖：
- 新鲜召回
- 标签兴趣召回
- 分类偏好召回
- 运营推荐召回

---

## 7. 曝光日志

当前推荐系统已经记录推荐曝光日志，包括：
- `videoId`
- `score`
- `channels`
- `strategyVersion`

其核心价值是帮助回答：
- 某个视频为什么会被推荐
- 它来自哪些召回通道
- 当前策略版本下不同通道的效果如何

这让推荐系统具备了更强的可分析性和可解释性，而不是一个纯黑盒排序流程。

---

## 8. 当前已落地能力

当前已经落地并可对外表达的推荐能力包括：

### 8.1 主链路
- 多路召回
- 线性打分
- 页面级重排
- 曝光日志落库

### 8.2 推荐输入层
- 用户兴趣标签画像
- 视频标签特征
- 最近观看惩罚
- 分类偏好与作者偏好

### 8.3 与缓存体系的关系
当前推荐主链路已经与现有内容层缓存体系相衔接：
- 视频详情内容层采用 `video:base:v2 + video:stat:{id}`
- 推荐结果本身正在推进独立的结果层缓存设计

### 8.4 与测试验证的关系
当前推荐系统不只是“能跑”，还已经具备了基础验证工具链：
- `seed generator` 可以批量生成用户、视频、行为与推荐测试数据
- `behavior simulator` 可以通过现有 HTTP API 注入播放、点赞、收藏、关注行为
- 可直接观察不同画像用户的推荐流变化，以及冷启动与热门结果差异

---

## 9. 当前边界与后续演进

### 9.1 已设计 / 正在推进
当前推荐系统除了已落地主链路外，下一步重点优化方向是：

- 推荐结果缓存
- 更严格的热点 key 重建保护
- 更丰富的画像特征
- 推荐反馈闭环

### 9.2 推荐结果缓存的设计方向
推荐结果缓存当前的思路不是缓存完整 `VideoVO`，而是：

- 缓存短时间稳定的推荐结果窗口 `videoIds`
- 游客和登录用户分 key
- 使用短 TTL
- 命中缓存后继续复用现有内容层缓存和 `VideoViewAssembler`

这样做的好处是：
- 不让推荐缓存和详情缓存耦合
- 避免缓存完整 VO 带来的失效复杂度
- 让推荐结果短时间稳定，但不会长期固定不变

### 9.3 仍未完全落地的内容
当前更适合定义为“下一步方向”的内容包括：
- 兴趣标签时间衰减
- 更多负反馈信号
- 曝光后点击/完播率回流
- 推荐效果指标统计（CTR、完播率、收藏率）
- 更完整的推荐结果缓存稳定化

## 10. 当前推荐系统怎么验证

### 10.1 冷启动验证
通过 seed 生成新用户或使用 cold_start 行为模板，不注入任何行为，观察推荐结果是否主要来自：
- 热门召回
- 新鲜召回
- 运营推荐召回

### 10.2 画像验证
使用 behavior simulator 构造 technology / food / lifestyle 等不同画像用户，批量模拟：
- 播放
- 进度上报
- 点赞
- 收藏
- 关注

然后分别请求 `/video/recommended`，观察推荐结果是否向对应兴趣偏移。

### 10.3 热门验证
行为模拟后可以同时查看：
- `/video/hot`
- Redis `video:hot:24h`

观察单个账号或多账号行为对热榜的影响。

## 11. 面试怎么讲

如果在面试里讲推荐系统，我建议把它分成四层说：

### 第一层：为什么不能只做热门榜
只靠热门榜会让内容越来越单一，新视频也很难获得曝光，所以推荐必须引入更多候选来源。

### 第二层：为什么拆成召回 / 打分 / 重排
- 召回负责覆盖面
- 打分负责相关性
- 重排负责页面观感

这三层分工不同，拆开后更容易调优，也更容易向面试官解释清楚推荐链路。

### 第三层：为什么当前阶段用线性模型
线性模型的优点是：
- 可解释
- 可调权
- 可快速验证
- 适合项目当前阶段落地

它不是最终形态，但非常适合作为推荐系统 V1 的稳定基础。

### 第四层：为什么要补验证工具链
很多项目只能“写出推荐接口”，但很难系统验证推荐是否真的受用户行为影响。这个项目通过 seed 数据生成和 behavior simulator，把推荐从“能跑”进一步推进到了“可重复验证”。

---

## 12. 总结

当前推荐系统的核心原则可以概括为：

- 先用 backend 自己的数据把推荐链路做完整
- 多路召回保证覆盖面
- 特征打分保证相关性
- 页面级重排保证观感
- 曝光日志保证可分析性
- seed + behavior simulator 保证验证能力
- 结果层缓存逐步补齐性能优化闭环

这让当前版本的推荐系统不仅具备基本效果，也具备后续继续增强和展开系统设计讨论的基础。
