# Video-Platform 系统架构说明

## 1. 项目整体定位

本项目是一个面向视频内容分发场景的全栈系统，核心目标是实现视频上传、播放、互动、搜索、消息和智能问答等能力。整体采用前后端分离架构，并通过独立的 Python 服务承载部分智能检索和生成能力。

系统当前由以下三个主要模块组成：

- `backend`：Spring Boot 后端，负责业务逻辑、数据访问、鉴权、缓存、对象存储、搜索、消息队列等
- `frontend`：Vue 3 前端，负责页面展示、用户交互和接口调用
- `agent-service`：Python 服务，负责智能问答、任务轮询和与大模型相关的辅助能力

## 2. 总体架构

```text
用户浏览器
   |
   v
Vue3 前端
   |
   v
Spring Boot 后端
   |
   |-- MySQL：业务主数据
   |-- Redis：缓存、状态数据、热点数据
   |-- Elasticsearch：搜索索引
   |-- MinIO：视频、封面、头像等对象存储
   |-- RocketMQ：异步任务、解耦处理
   |
   v
Python agent-service：智能问答 / 检索增强能力
```

## 3. 后端核心分层

后端整体采用典型的分层结构，主要包括：

- `controller`：对外暴露 REST 接口，处理参数接收、结果返回
- `service`：承载业务逻辑，协调数据库、缓存、消息队列等基础设施
- `mapper`：基于 MyBatis-Plus 的数据访问层
- `entity`：数据库实体对象
- `model/vo`：接口返回对象，用于页面展示数据封装
- `config`：Spring、Redis、跨域、拦截器等配置
- `mq`：异步消息生产与消费逻辑
- `utils`：JWT、上下文、工具类等通用能力

## 4. 主要业务域

### 4.1 用户域

负责用户注册、登录、验证码、用户信息、创作者数据、头像更新等能力。

核心内容包括：

- 用户注册与登录
- JWT 鉴权
- 用户主页信息
- 创作者统计信息
- 头像上传和访问

### 4.2 视频域

这是系统最核心的业务域，负责视频上传、视频详情、推荐列表、热门列表、创作者作品、播放统计、观看历史等能力。

当前视频域已经做了职责拆分：

- `VideoServiceImpl`：作为门面层，对外保持统一调用入口
- `VideoQueryService`：负责视频查询、列表聚合、详情查询
- `VideoCommandService`：负责上传、删除、播放记录、进度保存等写操作
- `VideoViewAssembler`：负责把视频实体和附属数据组装为页面展示对象

这种拆分的目标是减少单个 Service 类的职责膨胀，提高代码可维护性。

### 4.3 互动域

负责视频点赞、收藏、评论、弹幕、关注等互动能力。

这部分数据与视频详情、视频列表高度关联，因此在列表聚合阶段会统一批量加载互动相关状态，例如：

- 评论数
- 当前用户点赞状态
- 当前用户收藏状态
- 当前用户观看进度

### 4.4 消息域

负责私信会话、通知、系统消息和消息中心汇总。

消息域的核心目标是：

- 提供用户之间的私信通信能力
- 提供互动通知和系统通知能力
- 为前端消息侧栏和消息中心页面提供数据支持

### 4.5 搜索域

负责视频搜索、用户搜索、热门搜索和搜索历史等能力。

搜索域与 Elasticsearch 结合，主要目的是提升视频检索效率和搜索体验。

### 4.6 智能服务域

由 `agent-service` 承担，主要提供智能问答、检索任务、轮询结果等能力。

这部分与主后端通过 HTTP 调用衔接，使智能功能可以独立演进，而不会与 Spring Boot 主业务强耦合。

## 5. 核心数据流

### 5.1 视频上传链路

```text
前端上传视频/封面
   ->
后端接收文件
   ->
上传到 MinIO
   ->
写入 MySQL 视频记录
   ->
发送 MQ 消息
   ->
异步处理封面/视频相关后续任务
   ->
失效缓存并刷新查询结果
```

### 5.2 视频列表查询链路

```text
前端请求视频列表
   ->
后端查询视频主列表
   ->
批量加载作者、评论数、点赞状态、收藏状态、观看进度、统计值
   ->
VideoViewAssembler 组装 VideoVO
   ->
返回前端展示
```

### 5.3 视频详情查询链路

```text
前端请求视频详情
   ->
后端优先查询本地缓存/Redis
   ->
缓存未命中时回源数据库
   ->
组装详情 VO
   ->
写入缓存
   ->
返回前端
```

### 5.4 搜索链路

```text
前端提交关键词
   ->
后端调用搜索服务
   ->
Elasticsearch 检索候选视频
   ->
返回搜索结果
```

## 6. 关键设计点

### 6.1 前后端分离

前端与后端通过 REST API 通信，后端专注业务和数据，前端专注页面与交互。这样便于模块独立开发、联调和部署。

### 6.2 文件与业务数据分离

视频文件、封面和头像不直接存入数据库，而是存储在 MinIO 中；数据库只保存对象路径和业务元数据。这种方式更适合视频类场景，能够降低数据库存储压力。

### 6.3 查询与写入职责分离

视频模块在重构后，查询逻辑与写操作逻辑不再完全堆积在同一个 Service 中，有利于控制复杂度。

### 6.4 异步解耦

通过 RocketMQ 将部分处理流程异步化，例如视频处理完成后的缓存失效、后续任务触发等，降低接口同步阻塞时间。

### 6.5 搜索与主库分离

搜索能力交给 Elasticsearch 处理，避免主库承担复杂搜索压力，也使搜索排序、模糊匹配等能力更容易扩展。

## 7. 当前架构的优势

- 业务链路已经较完整，覆盖上传、播放、互动、搜索、消息等核心流程
- 技术栈较丰富，具备缓存、搜索、对象存储、消息队列等后端常见能力
- 已开始对核心模块做结构性重构，具备继续演进的基础

## 8. 当前架构仍可优化的方向

### 8.1 模块边界进一步清晰化

虽然视频模块已经做了实现层拆分，但接口层和业务域边界还可以进一步细化，例如后续拆分为：

- 视频查询接口
- 视频写操作接口
- 视频统计接口

### 8.2 工程化能力提升

可以继续补充：

- 配置外置化
- 统一异常处理
- 日志与可观测性
- 测试体系
- CI/CD

### 8.3 搜索与缓存策略持续优化

随着数据量增长，可以进一步加强：

- Redis 批量读取
- 统计值预聚合
- 搜索索引异步同步

## 9. 面试可重点讲解的架构亮点

在面试场景中，建议重点围绕以下内容展开：

- 视频平台为什么要使用 MinIO 存储大文件
- 为什么引入 Redis 做热点缓存和状态缓存
- 为什么使用 Elasticsearch 做搜索
- 为什么使用 RocketMQ 做异步解耦
- 视频列表查询如何从 N+1 查询优化为批量聚合查询
- 视频服务为什么要从大而全的 ServiceImpl 中拆分职责

## 10. 总结

本项目当前已经具备一个中小型视频平台的基础架构形态。虽然仍有工程化和高并发场景下的优化空间，但整体上已经覆盖了后端常见的核心技术组件，并且围绕视频列表聚合、缓存设计、对象存储、异步处理等典型问题做出了针对性的设计与优化。
