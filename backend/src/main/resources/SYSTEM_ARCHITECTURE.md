# Video-Platform 系统架构说明

## 1. 项目整体定位

本项目是一个面向视频内容平台场景的全栈系统，核心目标是实现视频上传、播放、互动、搜索、推荐、私信与通知等能力。整体采用前后端分离架构，后端为 **纯 Java** 实现，不依赖独立 Python agent 服务。

系统当前由两个主要模块组成：

- `backend`：Spring Boot 后端，负责业务逻辑、数据访问、鉴权、缓存、对象存储、搜索、推荐、消息队列等
- `frontend`：Vue 3 前端，负责页面展示、用户交互和接口调用

当前版本更适合定位为：

- **高完成度个人工程项目 / 开源展示项目**
- **适合学习、演示和面试讲解的 V1 版本**
- **已完成核心业务链路，具备继续增强的系统基础**

## 2. 总体架构

```text
用户浏览器
   |
   v
Vue 3 前端
   |
   v
Spring Boot 后端
   |
   |-- MySQL：业务主数据
   |-- Redis：缓存、状态数据、热点数据
   |-- Elasticsearch：搜索索引
   |-- MinIO：视频、封面、头像、消息图片等对象存储
   |-- RocketMQ：异步任务、解耦处理
```

## 3. 后端核心分层

- `controller`：对外暴露 REST 接口，处理参数接收与结果返回
- `service`：承载业务逻辑，协调数据库、缓存、搜索、消息队列等基础设施
- `mapper`：基于 MyBatis-Plus 的数据访问层
- `entity`：数据库实体对象
- `model/vo`：接口返回对象，用于页面展示数据封装
- `config`：Spring、Redis、跨域、拦截器等配置
- `mq`：异步消息生产与消费逻辑
- `utils`：JWT、上下文、工具类等通用能力
- `seed`：seed 数据生成器与推荐行为验证支撑能力

## 4. 主要业务域

### 4.1 用户域
- 用户注册与登录
- JWT 鉴权
- 用户主页与创作者信息
- 头像上传与访问

### 4.2 视频域
- 视频上传、删除、详情、播放统计、观看历史
- `VideoServiceImpl` 作为门面层
- `VideoQueryService` 负责查询类逻辑
- `VideoCommandService` 负责上传、删除、播放记录等写操作
- `VideoViewAssembler` 负责聚合并组装展示对象

### 4.3 互动域
- 点赞、收藏、评论、弹幕、关注等互动能力
- 列表聚合阶段批量加载互动状态与统计数据

### 4.4 消息域
- 私信会话
- 通知与系统消息
- 消息中心汇总与已读状态维护
- WebSocket 实时推送

### 4.5 搜索域
- Elasticsearch 关键词搜索（标题 / 描述）
- 用户搜索、热门搜索、搜索历史
- `/search/hybrid` 接口保留，当前实现为 ES-only
- 搜索索引异步同步与管理员触发的全量重建

### 4.6 推荐域
- backend-only 推荐系统
- 多路召回
- 线性打分
- 页面级重排
- 曝光日志记录
- 推荐结果缓存方案设计与实现推进中

### 4.7 内容补全与规则增强域
- 基于后端本地规则的标签推荐
- 投稿时标题、标签、分类的自动补全
- 基于行为和规则维护视频标签特征与推荐画像

### 4.8 测试与验证辅助域
- `seed` 数据生成器：批量生成用户、视频、互动与推荐测试数据
- `behavior simulator`：通过现有 HTTP API 批量模拟播放、进度、点赞、收藏、关注，并输出推荐 / 热门摘要

## 5. 核心数据流

### 5.1 视频上传链路

```text
前端上传视频/封面
   ->
后端接收文件
   ->
上传到 MinIO
   ->
写入 MySQL 视频记录
   ->
发送 MQ 消息
   ->
异步处理封面/视频相关后续任务
   ->
失效缓存并刷新查询结果
```

### 5.2 视频列表查询链路

```text
前端请求视频列表
   ->
后端查询视频主列表
   ->
批量加载作者、评论数、点赞状态、收藏状态、观看进度、统计值
   ->
VideoViewAssembler 组装 VideoVO
   ->
返回前端展示
```

### 5.3 视频详情读取链路

```text
前端请求视频详情
   ->
Caffeine（仅热点视频）
   ->
Redis base + stat
   ->
数据库 / 聚合逻辑
   ->
返回完整 VideoVO
```

### 5.4 搜索链路

```text
前端发起搜索
   ->
SearchServiceImpl
   ->
Elasticsearch 查询 title / description
   ->
补查业务数据
   ->
组装结果返回
```

### 5.5 推荐链路

```text
用户进入推荐页
   ->
多路召回
   ->
特征打分
   ->
页面级重排
   ->
曝光日志记录
   ->
返回推荐结果
```

### 5.6 推荐验证链路

```text
seed 生成测试用户/视频/行为
   ->
（可选）全量重建搜索索引
   ->
行为模拟脚本通过 HTTP API 注入播放/点赞/收藏/关注
   ->
请求推荐流 / 热榜
   ->
观察推荐偏移、多样性与热门结果变化
```

## 6. 当前工程亮点

### 6.1 缓存架构
- 视频详情使用 Caffeine + Redis 两级缓存
- 本地缓存已收敛为热点视频本地缓存
- 视频详情缓存正在从整对象缓存演进到 `base + stats` 分层

### 6.2 异步解耦
- RocketMQ 用于视频处理、封面处理、搜索同步、通知下发、资源清理、私信/系统通知推送
- 主链路只承担核心状态更新，后置逻辑通过消息异步化
- 当前 MQ 可靠性目标为：至少一次投递、消费端幂等保证业务效果接近 exactly-once、通过补偿机制实现最终一致性，不追求跨 DB 与 MQ 的全局强一致
- 当前 MQ 实现已补齐统一消息元数据（eventId / bizKey / traceId / eventType / version / occurredAt）、消息级/业务级幂等、producer dead-letter 补偿雏形、异常分类、消费失败台账与统一日志结构

### 6.3 搜索与推荐
- 搜索与推荐均由 backend 内部承载，不依赖独立 Python 服务
- 推荐系统已完成召回、打分、重排与曝光日志主链路
- seed generator 与 behavior simulator 让推荐系统具备了更完整的验证能力，而不是只停留在“功能存在”层面
- 搜索和推荐的结果层缓存设计已明确，正逐步推进实现

### 6.4 消息与通知闭环
- 消息中心、通知落库、未读数、WebSocket 推送已经形成闭环
- MQ 消费端已透传 consumerGroup，并在终态失败场景下写入消费失败台账，便于后续排查与人工介入
