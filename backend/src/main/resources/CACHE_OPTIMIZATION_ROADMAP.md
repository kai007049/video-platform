# Video-Platform 缓存优化路线图

## 1. 文档目的

这份文档聚焦于 **当前项目缓存体系的下一步优化方向**，目标不是推翻现有实现，而是回答三个问题：

1. 当前缓存已经覆盖了哪些核心场景？
2. 还有哪些地方值得继续补缓存？
3. 如果未来扩展到多实例部署，缓存架构应该如何演进？

它适用于：

- 后续开发迭代时的缓存优化参考
- README / 面试中解释缓存设计取舍
- 从“可运行”向“更工程化”演进时的路线图

---

## 2. 当前缓存现状

当前项目已经具备一套基础但不算简单的缓存体系，主要包括：

### 2.1 视频详情两级缓存

当前视频详情采用：

```text
Caffeine（仅热点视频详情） -> Redis -> DB/组装逻辑
```

已具备的能力：

- 本地缓存（Caffeine）
- Redis 详情缓存
- 空值缓存（防穿透）
- 分布式锁（防击穿）
- TTL 抖动（防雪崩）
- 主动失效 + 双删策略
- **本地缓存准入收敛：仅热榜 `video:hot:24h` Top 500 视频允许进入 Caffeine**

这部分是当前项目里最完整、最工程化的缓存落点。
### 2.2 元数据缓存

已经缓存：

- 分类树
- 分类列表
- 标签列表

这些对象具备“低频变更、高频读取”的特点，非常适合作为 Redis 元数据缓存。

### 2.3 搜索辅助缓存

已经缓存：

- 搜索历史
- 热搜关键词聚合结果

当前缓存主要聚焦于搜索行为和统计，而不是搜索结果本身。

### 2.4 未读数缓存

已经缓存：

- 私信未读数
- 通知未读数

这是典型的“高频读取、低复杂度对象”缓存场景。

---

## 3. 当前缓存体系的主要不足

虽然现有方案已经具备基础价值，但从后续扩展视角看，仍存在几个明显问题。

### 3.1 本地缓存使用范围已完成第一阶段收敛

当前视频详情缓存已经不再是“任意被访问过的视频都进入本地 Caffeine”。现阶段实现已调整为：**只有热榜 `video:hot:24h` Top 500 视频，才允许回填或写入本地缓存**。

这意味着第一阶段的热点本地缓存策略已经落地，主要收益是：

- 本地缓存一致性问题范围被收敛到热点视频
- 长尾数据不再持续污染本地缓存
- Redis 更明确地承担统一主缓存层职责

但从多实例演进角度看，后续仍可继续优化：

- 每次详情访问仍需查询 Redis 热榜资格
- 热门集合本身尚未做本地短 TTL 快照
- 热点判定目前仍是固定 Top N，而不是动态频次晋升

### 3.2 聚合对象缓存粒度较粗

当前缓存对象主要是完整 `VideoVO`，其中混合了：

- 视频基础信息
- 作者信息
- 统计信息
- 用户态信息

这种聚合缓存实现简单，但会带来：

- 某个字段变更时只能整对象失效
- 更新路径粗糙
- 后续难以做更精细的缓存拆分

### 3.3 高计算链路没有缓存结果

当前这些链路虽然开销较高，但尚未真正缓存结果：

- 推荐结果
- 搜索结果
- 用户画像摘要 / 推荐输入特征
- 会话列表 / 消息中心摘要

这些恰恰是最有缓存收益的地方。

### 3.4 缺乏缓存命中率视角

当前项目有缓存实现，但还没有形成以下观测指标：

- 本地缓存命中率
- Redis 命中率
- 回源数据库次数
- 推荐结果缓存命中率
- 搜索结果缓存命中率

这意味着当前缓存更偏“已经加了”，但还不是“可量化优化”。

---

## 4. 缓存优化的核心原则

后续优化建议遵循以下原则。

### 4.1 Redis 做统一主缓存层

在未来多实例场景中，Redis 更适合作为统一主缓存层。

本地缓存的职责应逐渐收敛为：

- 热点加速层
- 极高频详情兜底层
- 降低 Redis 网络开销的辅助层

### 4.2 优先缓存 ID 或摘要，而不是完整大对象

对于推荐和搜索这类结果型场景，优先缓存：

- videoId 列表
- 排序结果
- 摘要信息

而不是直接缓存完整 `VideoVO`。

原因：

- 失效更简单
- 可复用现有组装逻辑
- 更适合多处复用

### 4.3 高计算链路优先于静态链路

后续缓存新增优先级，应该优先考虑：

1. 计算贵
2. 重复请求多
3. 结果短时间内稳定

因此推荐、搜索、用户画像，比继续缓存更多静态表更值得优先做。

### 4.4 失效策略必须与写路径绑定

任何新增缓存都必须同时明确：

- 谁负责写缓存
- 谁负责删缓存
- 哪些行为会触发失效
- 是否允许短 TTL 自然过期兜底

否则缓存越加越乱。

---

## 5. 推荐的缓存优化优先级

下面按 **P1 / P2 / P3** 给出推荐路线。

---

## 6. P1：最值得优先补的缓存点

### 6.1 推荐结果缓存

#### 当前问题

推荐流当前每次请求都会经历：

- 多路召回
- 用户兴趣提取
- 标签特征匹配
- 打分
- 重排
- 组装 VO

这条链路计算成本明显高于普通列表接口。

#### 建议做法

优先缓存：

- 游客推荐第一页结果 ID 列表
- 用户推荐结果的短 TTL ID 列表

建议 key 例子：

```text
rec:guest:page:1:size:12
rec:user:1001:page:1:size:12
```

缓存内容示例：

```json
[101, 205, 309, 411, 512]
```

#### 为什么优先缓存 ID 而不是整 VO

- VO 聚合字段多，失效复杂
- ID 列表更稳定
- 可以复用现有 `VideoViewAssembler`

#### 建议 TTL

- 游客推荐：30 秒 ~ 2 分钟
- 用户推荐：30 秒 ~ 1 分钟

#### 建议收益

- 降低重复计算
- 降低多路召回开销
- 提升首页推荐流响应速度

---

### 6.2 搜索结果缓存

#### 当前问题

当前搜索虽然缓存了：

- 热搜词
- 搜索历史

但真正的视频搜索结果没有缓存。热门关键词会重复触发：

- Elasticsearch 查询
- MySQL 视频补查
- 结果组装

#### 建议做法

缓存热门搜索词的结果 ID 列表。

建议 key 例子：

```text
search:video:java:1:12:default
search:video:nba:1:12:save
```

缓存内容：

- 命中的 videoId 列表
- 可选：总命中数

#### 建议 TTL

- 30 秒 ~ 3 分钟

#### 建议收益

- 降低 ES 压力
- 降低重复补查数据库成本
- 热门搜索体验更稳定

---

### 6.3 用户画像 / 推荐输入缓存

#### 当前问题

推荐系统的输入特征当前主要是动态查出来的，例如：

- Top interest tags
- 最近观看视频
- 最近点赞 / 收藏信号
- buildProfileSummary 输出字符串

这些数据更新频率并不高，但重复读取频率可能很高。

#### 建议做法

缓存：

```text
user:profile:summary:{userId}
user:recent:watched:{userId}
user:top:tags:{userId}
```

#### 建议 TTL

- 1 分钟 ~ 5 分钟

#### 失效触发点

这些行为发生后可主动删缓存：

- 点赞
- 收藏
- 评论
- 观看进度更新
- 关注行为（如果后续参与画像）

#### 建议收益

- 降低推荐前置特征查询成本
- 让推荐系统输入层更稳定
- 有利于后续构建更清晰的推荐架构

---

## 7. P2：中期值得补的缓存点

### 7.1 热门榜单 / 最新列表缓存

适合缓存：

- `hot:list:{page}:{size}`
- `latest:list:{page}:{size}`

#### 原因

- 首页 / 热门页访问频繁
- 短时间内变化有限
- 实现简单，收益直接

#### 建议 TTL

- 30 秒 ~ 2 分钟

---

### 7.2 消息中心摘要 / 会话列表缓存

当前未读数已经缓存，但摘要与会话列表还主要靠实时组装。

适合缓存：

- `msg:center:summary:{userId}`
- `msg:conversations:{userId}`

#### 建议策略

- TTL 短：20 秒 ~ 60 秒
- 行为触发删缓存：发消息、标已读、清空会话、通知写入

#### 价值

- 实现复杂度不高
- 对消息中心页面体验有帮助

---

### 7.3 创作者主页 / 用户主页列表缓存

适合缓存：

- `author:videos:{authorId}:{page}:{size}`
- `user:profile:{userId}`
- `creator:dashboard:list:{userId}:{page}:{size}`

#### 原因

- 这些页面读多写少
- 结构稳定
- 很适合作为展示型缓存

---

## 8. P3：架构层面的缓存演进

### 8.1 热点本地缓存的下一步演进

#### 当前状态

当前实现已经完成第一阶段收敛：

- Redis 仍是统一主缓存层
- 只有热榜 Top 500 视频详情允许进入本地 Caffeine
- 热度来源直接复用 `video:hot:24h`

#### 下一步可继续优化的方向

1. 给“热门视频 ID 集合”增加短 TTL 本地快照，避免每次详情都查 Redis 热榜 rank
2. 在 Top N 之外增加分数阈值，进一步收缩本地缓存集合
3. 逐步从“固定 Top N”演进到“基于访问频次的动态热点晋升”

#### 当前阶段的价值

这一版已经实现了：

```text
Redis = 主缓存
Caffeine = 热点加速层
```

因此后续这一节更关注“如何继续降低热点判定成本”，而不再是“是否应该先把本地缓存收敛为热点缓存”。

---

### 8.2 视频缓存拆粒度

#### 当前问题

完整 `VideoVO` 缓存粒度偏粗。

#### 建议方向

后续可拆分为：

- 视频基础详情缓存
- 统计值缓存
- 用户态缓存（liked/favorited/progress）

#### 价值

- 更适合高频统计更新
- 避免整对象频繁失效
- 有助于多页面复用

---

### 8.3 缓存一致性广播

当系统走向多实例后，可以考虑：

- Redis pub/sub 广播缓存失效
- MQ 广播缓存失效事件
- 热点 key 异步刷新

当前阶段不必马上做，但应作为多实例演进方向保留。

---

### 8.4 增加缓存命中与回源监控

建议后续至少补这几项指标：

- local cache hit / miss
- redis cache hit / miss
- DB fallback count
- recommend cache hit ratio
- search cache hit ratio

即便不接完整监控系统，也建议在代码层面预留日志或统计埋点。

---

## 9. 不建议当前阶段优先缓存的场景

### 9.1 私信历史分页

原因：

- 变动频繁
- 已读 / 删除 / 撤回干扰大
- 缓存一致性复杂

当前阶段优先级不高。

### 9.2 再缓存更多静态表

分类和标签已经缓存到位。继续缓存更多“几乎不变的小表”，边际收益通常不如推荐、搜索和画像缓存。

### 9.3 在多实例前提下把本地缓存铺满更多列表场景

如果未来真的打算扩展多实例，本地缓存应该更谨慎，而不是越铺越广。

---

## 10. 建议的实施顺序

### 第一阶段（最实用）

1. 热点本地缓存收敛（已完成：仅热榜 Top 500 视频进入 Caffeine）
2. 推荐结果 ID 列表缓存
3. 搜索结果 ID 列表缓存
4. 用户画像摘要缓存

### 第二阶段（体验增强）

4. 热门 / 最新列表缓存
5. 消息中心摘要与会话列表短 TTL 缓存
6. 创作者主页列表缓存

### 第三阶段（架构升级）

7. 本地缓存仅保留热点视频
8. 视频缓存拆粒度
9. 多实例失效广播
10. 缓存命中率监控

---

## 11. 面试可讲的缓存优化思路

可以这样概括项目的缓存设计与下一步演进：

### 11.1 当前版本

- 已对视频详情做 Redis + Caffeine 两级缓存
- 已缓存分类树、标签列表、热搜词、搜索历史、未读数等高频基础数据
- 已实现空值缓存、分布式锁、TTL 抖动和主动失效等基础策略

### 11.2 下一步优化重点

- 对推荐结果做短 TTL 结果缓存
- 对热门搜索结果做分页缓存
- 对用户画像输入做摘要缓存
- 当前已完成“Redis 作为统一主缓存层、本地缓存收敛为热点详情缓存”的第一阶段改造
- 下一步可继续优化热点判定成本，例如给热门集合增加短 TTL 本地快照

### 11.3 多实例场景下的演进

- 当前两级缓存适合单实例与项目现阶段
- 若扩展为多实例，本地缓存应只保留热点 key
- 更长期可通过广播失效或事件通知解决多实例一致性问题

---

## 12. 总结

当前项目的缓存体系已经覆盖了部分核心高频读取场景，但从推荐、搜索和多实例演进的角度看，仍有较大优化空间。

最值得优先补的不是更多静态元数据缓存，而是：

- 推荐结果缓存
- 搜索结果缓存
- 用户画像 / 推荐输入缓存

而从更长期的架构演进看，最重要的方向是：

> **Redis 作为统一主缓存层，本地缓存收敛为热点加速层，缓存对象逐步从“大而全”向“分层、可失效、可观测”演进。**
