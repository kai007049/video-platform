# 搜索结果缓存实现计划

## 1. 文档目的

这份文档用于明确 **视频搜索结果缓存** 的首版实施方案，目标是在不破坏现有搜索功能的前提下，为高频搜索请求增加结果层缓存，降低 Elasticsearch 与数据库重复回源成本。

当前项目已经具备：

- 视频详情内容层缓存：`video:base:v2:{id}` + `video:stat:{id}`
- 搜索历史缓存
- 热搜词缓存

但还没有真正的：

- `query -> ids + total` 级别的搜索结果缓存

因此本次设计聚焦于：

> 用最小复杂度补齐“视频搜索结果缓存”，优先缓存高频、短时间稳定、重复请求多的第一页搜索结果。

---

## 2. 设计目标

首版搜索结果缓存要满足这些目标：

1. 降低热门关键词搜索时的 ES 重复查询。
2. 降低热门关键词搜索结果对数据库补查和组装的重复开销。
3. 保持结果层与内容层分离，不缓存完整 `VideoVO`。
4. 避免搜索 key 空间失控，因此必须做 query 规范化和缓存准入控制。
5. 保证空结果也能被短暂缓存，减少无意义重复查询。

---

## 3. 为什么搜索结果缓存值得做

当前视频搜索链路是：

```text
前端搜索请求
  -> SearchServiceImpl.searchVideos()
  -> Elasticsearch 查询
  -> MySQL 查视频详情
  -> 组装返回
```

这里已经存在两个问题：

### 3.1 重复查询 ES
热门 query（例如 `java`、`springboot`、`nba`）可能在短时间内被大量重复搜索，但当前每次都要重新打 Elasticsearch。

### 3.2 重复补查数据库
搜索结果命中后，还需要再从 MySQL 查视频并组装结果，这一层也存在重复开销。

因此，搜索结果缓存属于典型的：

- 计算成本不低
- 重复请求明显
- 结果短时间相对稳定

非常适合作为结果层缓存的候选对象。

---

## 4. 缓存对象设计

### 4.1 缓存什么
首版建议缓存：

```text
query 对应的 videoIds + total + generatedAt
```

而不是：

- 完整 `VideoVO`
- 原始 ES 文档对象

### 4.2 缓存 value 结构

```json
{
  "ids": [101, 205, 309],
  "total": 278,
  "generatedAt": 1710000000
}
```

### 字段说明

#### `ids`
当前 query 对应的视频 ID 列表。

#### `total`
表示当前 **normalizedKeyword + sortBy + page/size 之外所有已参与搜索结果计算的条件组合下**，搜索引擎返回的总命中数，用于前端分页或结果数量展示。

#### `generatedAt`
本次搜索结果缓存生成的时间戳，建议统一使用：

- **Unix 时间戳（秒）**

作用：
- 帮助排查结果是什么时候生成的
- 为后续软过期策略预留能力
- 便于分析某批结果被复用了多久

---

## 5. key 设计

首版建议采用 **page key**，不做 window 化。

### key 模板

```text
search:video:{normalizedKeyword}:sort:{sortBy}:page:{page}:size:{size}:v1
```

例如：

```text
search:video:java:sort:comprehensive:page:1:size:20:v1
```

### 为什么搜索首版不用 window key

和推荐不同，搜索分页本来就是用户显式请求的一部分，因此 page 级 key 更自然：

- query 参数天然决定一次查询语义
- page/size 是搜索结果的一部分
- 没必要为了统一风格强行把搜索也改成 window

所以搜索结果缓存首版继续用 page key 是合理的。

---

## 6. query 规范化（必须做）

这是搜索结果缓存最关键的前置条件之一。

### 为什么必须做
如果不规范化 query，就会出现很多语义接近但 key 不同的缓存：

- `Java`
- `java`
- ` java `
- `springboot`
- `spring boot`

这样会严重稀释命中率。

### 首版最少做的规范化

- `trim`
- `lowercase`
- 多空格压缩为单空格

### 后续可选增强

- 全角/半角统一
- 常见技术词归一化

### 设计要求

缓存 key 必须使用 **规范化后的 query**，而不是原始输入。

### 边界条件

- 规范化后 query 不能为空
- 明显无效的异常符号 query 不进入结果缓存
- 如果业务层后续定义了“非法搜索词”规则，应优先遵循业务规则，再决定是否进入缓存

---

## 7. 缓存准入控制（必须做）

搜索结果缓存和推荐结果缓存最大的不同在于：

> 搜索的 key 空间天然更大，不能默认缓存所有 query。

### 首版建议准入规则

只缓存满足以下条件的请求：

1. `page = 1`
2. `size` 是常见值（例如 `10` / `12` / `20`）
3. `normalizedKeyword` 长度在合理范围内（例如 `2 ~ 20`）
4. query 规范化后不能为空，且不属于明显无效 query

### 关于“高频 query”的首版定义

首版**不严格依赖真实的 query 热度统计**，而是通过：

- 第一页
- 常见 size
- query 长度限制
- 无效 query 过滤

来近似控制缓存范围。

也就是说，首版更准确的表述不是“只缓存真正高频 query”，而是：

> **先缓存符合第一页 + 常见 size + query 长度限制等条件的高收益 query 请求，作为搜索结果缓存的第一阶段实现。**

后续如果要更严格地做“高频 query 缓存”，再补：

- `search:query:freq:{normalizedKeyword}`
- 基于时间窗口的 query 访问计数
- 达到阈值后才允许写入搜索结果缓存

### 为什么先只缓存第一页
因为：

- 大多数搜索行为集中在第一页
- 第一页命中率和收益最高
- 能有效控制 key 数量

### 当前阶段不建议做
- 缓存所有页
- 缓存所有 query
- 缓存超长 query

---

## 8. 空结果缓存

空结果同样值得缓存。

### 结构示例

```json
{
  "ids": [],
  "total": 0,
  "generatedAt": 1710000000
}
```

### 为什么值得缓存
某些热门误拼 query 或者冷门 query，如果不缓存空结果，会导致 ES 被反复查询。

### TTL 建议
空结果 TTL 可以比正常结果更短：

- `30 秒`

这样既降低重复无效查询，又避免长期缓存错误拼写结果。

---

## 9. TTL 设计

### 正常结果
- 建议 TTL：`30 ~ 180 秒`
- 首版推荐：`60 秒`

### 空结果
- 建议 TTL：`30 秒`

### 原因
搜索结果通常比推荐结果更稳定，但由于 key 空间更大，因此：

- TTL 可以稍长一点
- 但准入必须更严格

---

## 10. 读写路径设计

### 命中缓存时

```text
请求搜索
  -> 先对 keyword 做规范化
  -> 判断是否满足缓存准入规则
  -> 查搜索结果缓存
  -> 命中后拿到 ids + total
  -> 按 ids 保序加载视频
  -> 若部分视频已不可展示或组装失败，则在当前页结果中过滤
  -> 走现有内容层组装逻辑
  -> 返回结果
```

### 未命中缓存时

```text
请求搜索
  -> 先对 keyword 做规范化
  -> 判断是否满足缓存准入规则
  -> 查 Elasticsearch
  -> 查 MySQL / 组装 ids + total
  -> 若满足准入规则则写入缓存
  -> 返回结果
```

### 首版过滤策略

如果缓存命中后，部分 `ids` 对应的视频已经：

- 被删除
- 不可见
- 组装失败

首版建议：

- 在组装阶段直接过滤掉这些视频
- **不做复杂补位**
- 允许当前页实际返回数量小于 `page size`

这样实现更简单，也更符合当前项目阶段的复杂度控制要求。

---

## 11. 与现有缓存体系的配合方式

搜索结果缓存只负责：

- “这个 query 命中了哪些 videoIds”
- “total 是多少”

真正的视频内容展示仍然由现有内容层缓存承担：

- `video:base:v2:{id}`
- `video:stat:{id}`

这样搜索结果缓存与视频内容缓存职责分离，不互相污染。

---

## 12. 失效与击穿保护策略

### 首版建议
以 TTL 为主，不做复杂的主动精确失效。

### 原因
搜索结果本身不是强一致业务缓存，且当前项目已经有搜索索引同步链路，因此首版没必要把失效做得太复杂。

### 击穿保护建议
虽然搜索结果缓存没有推荐游客窗口那样强热点，但对于：

- `java`
- `springboot`
- `nba`

这类高频 query，一样可能出现缓存同时过期后的瞬时击穿。

因此首版建议至少考虑：

- TTL 加少量随机抖动
- 对热点 key 预留短暂重建锁 / single-flight 保护能力

即使首版不立即实现完整击穿保护，也建议把它作为明确风险写入设计中。

### 后续可选增强
如果以后确实需要更细致控制，可以考虑在这些场景下删热点 query 缓存：

- 视频标题变化
- 视频描述变化
- 视频删除
- 搜索索引重建后

但首版不建议先做这一步。

## 14. 实施步骤

### Step 1
在 `RedisConstants.java` 中新增：
- 搜索结果缓存 key 前缀
- 搜索结果缓存 TTL
- 空结果缓存 TTL

### Step 2
在 `SearchServiceImpl` 中增加 query 规范化函数：
- trim
- lowercase
- 多空格压缩
- 对规范化后为空的 query 直接走现有业务返回，不进入结果缓存

### Step 3
在 `SearchServiceImpl` 中增加缓存准入判断函数：
- 是否第一页
- 是否常见 size
- query 长度是否合理
- 是否属于明显无效 query

### Step 4
在 `searchVideos()` 中接入结果缓存读写：
- 命中时直接走缓存 ids + total
- 未命中时查 ES + DB
- 命中结果符合准入规则时写缓存
- 命中缓存后若部分视频不可展示，首版直接过滤，不做复杂补位

### Step 5
支持空结果缓存：
- `ids = []`
- `total = 0`
- `generatedAt = unix timestamp（秒）`
- TTL 更短

## 15. 验证方案

### 功能验证
- 同一个热门关键词第一页短时间内重复搜索，应命中缓存
- TTL 到期后，再次请求应触发重新查询
- 搜索结果仍然完整（封面、作者、统计值都正常）
- 空结果搜索短时间内重复查询，应命中空结果缓存

### key 验证
应能在 Redis 中看到类似 key：

```text
search:video:admin:sort:comprehensive:page:1:size:20:v1
```

### 边界验证
- `keyword` 大小写不同但语义相同，应命中同一条缓存
- 非第一页请求，默认不进结果缓存
- 长度异常的 query，不应进入缓存

---

## 16. 总结

首版搜索结果缓存的最合理实现路径是：

- 使用 **page key**
- 使用 **规范化 query**
- 只缓存 **第一页、高频、短时间稳定** 的搜索结果
- 缓存内容为：`ids + total + generatedAt`
- 支持空结果缓存
- 不缓存完整 `VideoVO`

这样既能明显提升高频搜索体验，又不会让搜索 key 空间和缓存失效复杂度失控。
